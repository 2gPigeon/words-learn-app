# 単語テスト・学習アプリ 設計書

## 1. 目的

本アプリは、ユーザーが単語帳を選択し、**学習**と**テスト**をスマホ・PC のブラウザ上で行えるようにすることを目的とする。
PWA として配布し、**ホーム画面追加**と**オフライン利用**を可能にする。
バックエンドは持たず、問題データは静的 JSON とし、ユーザーごとの学習履歴・進捗・設定はブラウザ内の IndexedDB に保存する。
Cloudflare Pages 上で配信し、低コストかつ簡易な運用を目指す。 ([vite-pwa-org.netlify.app][2])

---

## 2. 採用技術

### 2.1 フロントエンド

**推奨: Vite + React + TypeScript**

理由:

* Codex に書かせる前提なら、React + TypeScript はサンプル量が多く、部品分割・状態管理・フォーム実装が安定しやすい
* Vite は SPA の初期構築と本番ビルドが軽く、静的配信と相性が良い
* 将来のコンポーネント分割やテスト導入もやりやすい

補足:

* Vue / Svelte でも成立する
* ただし、**最初のプロトタイプ速度**と**Codex の書きやすさ**を優先して React を第一候補とする
* vite-plugin-pwa は Vue / React / Svelte を含む複数テンプレートに対応しているため、後から他フレームワークへ寄せることも可能 ([vite-pwa-org.netlify.app][2])

### 2.2 PWA

**vite-plugin-pwa**

利用目的:

* Web App Manifest の生成
* Service Worker の生成
* Service Worker の登録
* オフラインキャッシュ制御

初期方針:

* **`generateSW` を採用**
* 更新方式は **`prompt` を採用**

  * 学習中・解答中に自動更新が走ると、入力中データの体験を壊しやすいため
  * 将来的により細かいキャッシュ制御が必要になったら `injectManifest` へ拡張する ([vite-pwa-org.netlify.app][3])

### 2.3 データ保存

**IndexedDB**

利用目的:

* 学習履歴
* テスト結果
* 単語ごとの習熟度
* 最終学習日時
* アプリ設定

IndexedDB は構造化データの永続化に向いており、トランザクションとインデックスを持てるため、単語学習履歴のようなデータ管理に適している。オフライン時にも利用可能。 ([MDNウェブドキュメント][4])

### 2.4 配信

**Cloudflare Pages**

利用目的:

* 静的ファイル配信
* Git 連携デプロイ
* 独自ドメイン運用
* HTTPS 配信

Cloudflare Pages は Vite 系サイトをそのまま扱え、push ベースで継続デプロイできる。SPA ルーティングには `_redirects` を置く。 ([Cloudflare Docs][5])

## 7. データ設計

## 7.1 public 配下 JSON 設計

単語帳データは **公開前提**なので、秘密情報は含めない。
構成例:

```txt
public/
  decks/
    index.json
    basic-english.json
    toeic-500.json
    it-terms.json
```

### `public/decks/index.json`

```json
[
  {
    "id": "basic-english",
    "title": "Basic English",
    "description": "基本英単語",
    "version": 1,
    "itemCount": 120
  },
  {
    "id": "it-terms",
    "title": "IT Terms",
    "description": "IT用語集",
    "version": 1,
    "itemCount": 80
  }
]
```

### `public/decks/basic-english.json`

```json
{
  "id": "basic-english",
  "title": "Basic English",
  "version": 1,
  "lang": "en-ja",
  "items": [
    {
      "id": "basic-english-0001",
      "question": "apple",
      "answer": "りんご",
      "choices": ["みかん", "りんご", "ぶどう", "もも"],
      "tags": ["fruit"],
      "note": "基本単語"
    }
  ]
}
```

## 7.2 JSON の要件

* `id` は不変
* `version` を持つ
* 単語追加・変更時にバージョンを更新
* `choices` は任意
* `note` は任意
* `tags` は任意
* 後方互換を壊す変更時はメジャー扱いで管理

## 8. IndexedDB 設計

## 8.1 DB 名

```txt
word-learning-app-db
```

## 8.2 バージョン

```txt
version 1
```

将来のスキーマ変更に備えて version 管理を行う。IndexedDB はスキーマ変更を伴う更新時にバージョン管理が必要で、トランザクション前提で扱う。 ([MDNウェブドキュメント][7])

## 8.3 Object Store 一覧

### 1. `deckProgress`

単語帳単位の進捗

```ts
{
  deckId: string
  lastStudiedAt: number
  lastTestedAt: number
  totalStudied: number
  totalCorrect: number
  totalAnswered: number
}
```

Key:

```txt
deckId
```

### 2. `wordProgress`

単語単位の進捗

```ts
{
  deckId: string
  wordId: string
  familiarity: number      // 0-5
  correctCount: number
  wrongCount: number
  lastAnsweredAt: number | null
  nextReviewAt: number | null
  lastResult: "correct" | "wrong" | "skip" | null
}
```

Key:

```txt
[deckId, wordId]
```

Indexes:

```txt
deckId
nextReviewAt
familiarity
```

### 3. `sessions`

テストセッション単位

```ts
{
  sessionId: string
  deckId: string
  mode: "study" | "test"
  startedAt: number
  finishedAt: number | null
  total: number
  correct: number
  wrong: number
}
```

Key:

```txt
sessionId
```

Indexes:

```txt
deckId
startedAt
finishedAt
```

### 4. `answers`

回答ログ

```ts
{
  answerId: string
  sessionId: string
  deckId: string
  wordId: string
  userAnswer: string | null
  isCorrect: boolean
  answeredAt: number
  responseMs: number | null
}
```

Indexes:

```txt
sessionId
deckId
wordId
answeredAt
```

### 5. `settings`

設定情報

```ts
{
  key: string
  value: unknown
}
```

例:

* theme
* questionsPerTest
* testMode
* sortMode

## 9. 出題ロジック設計

## 9.1 学習モード

目的:

* 単語の確認
* 自己評価による習熟度更新

動作:

* 単語を順番またはランダムで表示
* ユーザーが「覚えた / あやしい / 苦手」で評価
* `familiarity` を更新
* `nextReviewAt` を更新

更新例:

* 覚えた → familiarity +1
* あやしい → 変化なし
* 苦手 → familiarity -1
* 下限 0 / 上限 5

## 9.2 テストモード

目的:

* 正誤ベースで定着確認

出題優先順位:

1. `nextReviewAt` が過去
2. `familiarity` が低い
3. `wrongCount` が多い
4. 未回答
5. その他ランダム

## 9.3 習熟度アルゴリズム（簡易）

初期値:

```txt
familiarity = 0
```

正解時:

* `correctCount += 1`
* `familiarity = min(5, familiarity + 1)`

不正解時:

* `wrongCount += 1`
* `familiarity = max(0, familiarity - 1)`

次回復習日時:

* familiarity 0 → 1日後
* familiarity 1 → 2日後
* familiarity 2 → 4日後
* familiarity 3 → 7日後
* familiarity 4 → 14日後
* familiarity 5 → 30日後

初期プロトタイプではこの簡易方式で十分。
FSRS のような高度な間隔反復は後続対応でよい。

## 12. PWA 設計

vite-plugin-pwa は、manifest 生成・service worker 生成・登録コード生成をまとめて提供する。Service Worker 戦略は `generateSW` と `injectManifest` を選べる。更新方式は `autoUpdate` と `prompt` があり、フォーム入力などデータ消失リスクがあるアプリでは `prompt` のほうが扱いやすい。 ([vite-pwa-org.netlify.app][2])

## 12.1 方針

### 初期採用

* strategy: `generateSW`
* registerType: `prompt`

### 理由

* プロトタイプ段階では設定が簡単
* 独自 Service Worker を書かずに済む
* 学習途中に自動リロードされない

## 12.2 キャッシュ対象

### precache

* `index.html`
* JS/CSS bundle
* アイコン
* `decks/index.json`
* 小さめの deck JSON

### runtime cache

* `/decks/*.json`
* 画像
* フォント

## 12.3 更新通知

新バージョン検出時:

* 画面下部にトースト表示
* 「更新する」ボタン押下で SW 更新
* 「あとで」も選べる

## 12.4 オフライン方針

オフラインでも以下を可能にする:

* すでに読み込んだ単語帳の学習
* 進捗保存
* 履歴表示

オフラインでは制限されるもの:

* 初回未読込の単語帳取得
* 新規配信された deck の取得

## 12.5 開発時 PWA

必要に応じて `devOptions.enabled = true` を使い、開発中も SW 挙動確認を行う。 vite-plugin-pwa は開発用の SW 動作確認オプションを持つ。 ([vite-pwa-org.netlify.app][2])

## 14. JSON 読み込み設計

## 14.1 読み込み対象

* 起動時に `decks/index.json`
* 単語帳選択時に `decks/:id.json`

## 14.2 バージョン管理

* deck JSON に `version`
* IndexedDB 側に `deckVersion`
* JSON 更新を検知したらローカル progress との整合性確認
* `wordId` が消えた場合は孤立レコードをクリーンアップ

## 14.3 エラーハンドリング

* JSON 読込失敗 → リトライ導線
* deck が存在しない → 404 相当画面
* schema 不正 → 開発環境では詳細表示、本番は簡易エラー

## 16. セキュリティ・運用上の注意

1. `public` 配下 JSON は誰でも取得できるため、秘密情報を含めない
2. バックエンドがないため、履歴は端末依存になる
3. ブラウザデータ削除時に履歴は消える
4. 複数端末同期はできない
5. 問題データ改ざん耐性は高くないため、厳密な試験用途には向かない

---

## 17. パフォーマンス方針

1. deck 一覧は初回ロードのみ
2. 単語帳本文は必要時ロード
3. 巨大 JSON は分割可能にしておく
4. 画像は極力使わない
5. 1 deck が極端に大きい場合は chunk 化を検討

## 21. 最終的な推奨構成

一番無難なのは、次の構成です。

* **Vite + React + TypeScript**
* **React Router**
* **vite-plugin-pwa**
* **IndexedDB**
* **public JSON**
* **Cloudflare Pages**

理由は、**実装の速さ・Codex の書きやすさ・静的配信との相性・PWA 化の容易さ**のバランスが最も良いからです。Vite は静的サイト向けビルドに向いており、vite-plugin-pwa は少ない設定で PWA 化でき、Cloudflare Pages はその成果物を継続デプロイしやすいです。 ([vitejs][1])
